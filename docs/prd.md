# rs-alias-manager 产品需求文档（PRD）

## 1. 项目信息

| 字段 | 内容 |
|------|------|
| 项目名称 | rs-alias-manager |
| 语言 | 中文 |
| 编程语言 | Rust（前端 + 后端，Tauri Rust 前端方案） |
| 框架 | Tauri（桌面应用） |
| 界面模板 | Vanilla（纯 Rust + HTML/CSS，不使用 React/Vue） |
| 原始需求 | 一个基于 Tauri 的系统别名管理器，支持系统 shell 别名的增删改查、常用别名模板一键导入、终端直接执行 |

---

## 2. 产品定义

### 2.1 产品目标

1. **简化别名管理**：为开发者提供可视化的 shell 别名管理界面，替代手动编辑 `.bashrc` / `.zshrc` 文件的方式，降低管理成本和出错风险
2. **加速环境搭建**：通过预置别名模板库，帮助开发者快速配置常用别名，减少重复劳动，尤其适合新机初始化场景
3. **即时生效体验**：别名变更后立即可在终端使用，无需手动 `source` 或重启终端，实现"所见即所得"的流畅体验

### 2.2 目标用户

- 日常使用终端的开发者（macOS / Linux）
- 频繁在多台机器间切换的开发者
- 刚接触命令行、不熟悉 shell 配置文件的新手开发者
- 追求效率、希望用别名优化工作流的资深开发者

### 2.3 用户故事

1. **As a** 开发者，**I want** 在可视化界面中查看当前所有 shell 别名，**so that** 我不用记住配置文件路径，也不用 `cat ~/.bashrc` 去翻找
2. **As a** 开发者，**I want** 通过表单添加/编辑/删除别名，**so that** 我不会因为语法错误（如漏写 `=` 或引号）导致配置文件失效
3. **As a** 新机初始化的开发者，**I want** 从预置模板库中一键导入常用别名（如 `gs='git status'`、`ll='ls -la'`），**so that** 我能在几分钟内搭建好熟悉的终端环境
4. **As a** 开发者，**I want** 添加别名后立即在终端中使用，**so that** 我不需要手动执行 `source ~/.bashrc` 或重启终端
5. **As a** 多 shell 用户，**I want** 应用自动识别我当前使用的 shell（bash/zsh）并管理对应的配置文件，**so that** 我不需要关心底层配置文件差异

## 3. 需求池

### P0 — 必须实现（MVP）

| 编号 | 需求 | 描述 |
|------|------|------|
| P0-1 | 别名列表展示 | 启动应用后自动读取当前 shell 的配置文件，解析并以列表形式展示所有别名及其对应命令 |
| P0-2 | 添加别名 | 用户可通过界面输入别名名称和命令，应用将其写入对应的 shell 配置文件（`~/.bashrc` 或 `~/.zshrc`） |
| P0-3 | 删除别名 | 用户可选中某条别名并删除，应用从配置文件中移除对应行 |
| P0-4 | 编辑别名 | 用户可修改已有别名的名称或命令，应用更新配置文件中对应内容 |
| P0-5 | Shell 类型自动检测 | 应用启动时自动检测用户默认 shell（通过 `SHELL` 环境变量），确定应读写哪个配置文件 |
| P0-6 | 配置文件安全写入 | 写入配置文件时采用"写临时文件 → 原子替换"策略，避免写入中断导致配置文件损坏 |
| P0-7 | 别名即时生效 | 别名变更后自动执行 `source` 操作（或通过 Tauri command 在后台 shell 中执行），使新别名在当前终端会话中立即可用 |

### P1 — 重要的增强功能

| 编号 | 需求 | 描述 |
|------|------|------|
| P1-1 | 别名模板库 | 提供预置的常用别名模板，分类展示（如 Git、文件操作、Docker、网络等），用户可预览并一键导入 |
| P1-2 | 模板一键导入 | 用户从模板库选择别名后，点击"导入"即可批量添加到配置文件，自动跳过已存在的同名别名并提示 |
| P1-3 | 搜索/过滤别名 | 支持按别名名称或命令内容搜索过滤，方便在别名较多时快速定位 |
| P1-4 | 别名分组/标签 | 支持为别名添加自定义标签或分组，按类别查看和管理别名 |
| P1-5 | 导出/导入配置 | 支持将当前别名配置导出为 JSON 文件，以及从 JSON 文件导入别名，便于跨机器迁移 |
| P1-6 | 多 Shell 支持 | 允许用户在应用中切换目标 shell（bash/zsh/fish），同时管理多种 shell 的配置文件 |

### P2 — 锦上添花

| 编号 | 需求 | 描述 |
|------|------|------|
| P2-1 | 别名使用统计 | 统计各别名的使用频率，帮助用户发现未使用的别名并清理 |
| P2-2 | 自定义模板 | 允许用户创建、保存和分享自己的别名模板集 |
| P2-3 | 别名冲突检测 | 添加别名时检测是否与系统命令或其他别名冲突，给出警告 |
| P2-4 | 配置文件变更监听 | 监听配置文件的外部变更（如用户在终端手动 `echo "alias ..." >> ~/.zshrc`），自动刷新界面 |
| P2-5 | 深色/浅色主题 | 支持界面主题切换 |
| P2-6 | 云同步 | 支持别名配置通过 Git 仓库或云存储同步 |

## 4. UI 设计稿

### 4.1 整体布局

应用采用单窗口布局，左侧为导航栏，右侧为主内容区。

```
+----------------------------------------------------------+
|  rs-alias-manager                              [_][□][×]  |
+----------+-----------------------------------------------+
|          |                                               |
| 导航栏    |              主内容区                          |
|          |                                               |
| 📋 别名  |  (根据左侧导航切换内容)                         |
| 📦 模板  |                                               |
| ⚙️ 设置  |                                               |
|          |                                               |
+----------+-----------------------------------------------+
```

### 4.2 别名管理页面（默认页）

```
+----------+-----------------------------------------------+
|          |  别名管理                              [搜索框] |
| 📋 别名  |                                               |
| (active) |  Shell: zsh (~/.zshrc)              [切换Shell] |
| 📦 模板  |                                               |
| ⚙️ 设置  |  +----+---------+---------------------------+   |
|          |  | ✓  | 别名    | 命令                      |   |
|          |  +----+---------+---------------------------+   |
|          |  | ☐  | gs      | git status                |   |
|          |  | ☐  | ll      | ls -la                    |   |
|          |  | ☐  | dc      | docker compose            |   |
|          |  | ☐  | gp      | git push                  |   |
|          |  +----+---------+---------------------------+   |
|          |                                               |
|          |  [＋添加别名]  [🗑删除选中]  [✏️编辑]           |
+----------+-----------------------------------------------+
```

**交互说明：**
- 列表默认展示当前 shell 的所有别名，按字母排序
- 点击行可选中，支持多选（用于批量删除）
- 双击行进入编辑模式
- 搜索框实时过滤列表
- 顶部显示当前 shell 类型和配置文件路径

### 4.3 添加/编辑别名弹窗

```
+-----------------------------------+
|  添加别名                    [×]  |
+-----------------------------------+
|                                   |
```

- "立即生效"默认勾选
- 编辑模式下弹窗标题改为"编辑别名"，按钮改为"确认保存"

## 5. 待确认问题

| 编号 | 问题 | 影响范围 | 建议 |
|------|------|----------|------|
| Q1 | Tauri Rust 前端方案的成熟度和可用 UI 组件库情况？Rust 前端方案（如 Dioxus、Leptos 等）的选型需要确认 | 技术可行性 | 建议在架构阶段做技术预研，确认 Tauri + Rust 前端的最佳实践 |
| Q2 | "别名即时生效"的实现方式：是通过 Tauri 后端执行 `source` 命令，还是写入配置文件后提示用户手动 source？注意 `source` 只影响子进程，无法影响用户已打开的其他终端窗口 | P0-7 的实现 | 建议：写入配置文件 + 弹出提示"新终端窗口将自动生效，已打开的终端请执行 source ~/.zshrc" |
| Q3 | 是否需要支持 fish shell？fish 的别名语法（`alias` / `abbr`）与 bash/zsh 不同，配置文件路径也不同（`~/.config/fish/config.fish`） | P1-6 的范围 | MVP 阶段建议仅支持 bash 和 zsh，fish 作为 P1 后续支持 |
| Q4 | 别名解析策略：用户的 `.bashrc` / `.zshrc` 中可能有 `if` 条件块、`source` 其他文件等复杂结构，别名可能散落在多个文件中。是否只解析主配置文件中的 `alias` 行？ | P0-1 的准确性 | MVP 建议只解析主配置文件（`~/.bashrc` / `~/.zshrc`）中以 `alias` 开头的行，不做递归解析 |
| Q5 | 配置文件中非别名内容的保护：用户配置文件中包含大量非别名内容（环境变量、PATH、函数定义等），写入时如何确保不破坏这些内容？ | P0-6 的安全性 | 采用"行级增删"策略：只新增 `alias xxx=yyy` 行、删除匹配的 alias 行、替换匹配的 alias 行，不触碰其他内容 |
| Q6 | 模板库的数据来源：预置模板是硬编码在应用内，还是从远程加载？是否支持用户贡献模板？ | P1-1 的实现 | MVP 硬编码内置模板即可，远程加载和用户贡献作为 P2 |
| Q7 | 目标操作系统：仅 macOS，还是同时支持 Linux？Windows 的 alias 机制完全不同（doskey / PowerShell Profile） | 项目范围 | 建议首版仅支持 macOS + Linux，不支持 Windows |

## 6. 技术约束备忘

| 约束 | 说明 |
|------|------|
| Tauri 版本 | 建议使用 Tauri v2（最新稳定版） |
| 前端方案 | Rust 前端（需确认具体框架：Dioxus / Leptos / Tauri 原生 webview + Rust-WASM） |
| 不使用 JS 框架 | 不使用 React / Vue / Svelte 等 JavaScript 框架 |
| 后端语言 | Rust（Tauri 后端天然 Rust） |
| CSS 方案 | Vanilla CSS 或 Tailwind CSS（待确认） |
